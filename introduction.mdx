CC Switch is a cross-platform desktop application that centralizes provider management, MCP servers, Skills, Sessions, and proxy failover for five AI coding tools: Claude Code, Codex, Gemini CLI, OpenCode, and OpenClaw. This page explains what CC Switch does, the problems it solves, and what you can do with it.

## What CC Switch solves

Modern AI-assisted development relies on CLI tools that each store their configuration in a different format and location. Switching API providers means manually editing JSON, TOML, or `.env` files — and there is no unified way to share MCP servers or Skills across tools.

CC Switch replaces all that manual editing with a single visual interface. You get one-click provider switching with 50+ built-in presets, unified MCP and Skills management across apps, and a local proxy with automatic failover — backed by a reliable SQLite database with atomic writes that protect your configs from corruption.

## How it works

CC Switch is built with Tauri 2 and Rust on the backend and React 18 + TypeScript on the frontend. All data — providers, MCP servers, prompts, and Skills — is stored in a local SQLite database at `~/.cc-switch/cc-switch.db`. Device-level settings are stored separately in `~/.cc-switch/settings.json`.

When you switch providers, CC Switch writes the new configuration directly to the corresponding CLI tool's config file using atomic writes (temp file + rename) to prevent corruption. For Claude Code, changes take effect immediately via hot-reload. For all other tools, you restart the terminal to pick up the new config.

<Note>
  Your API keys and provider configurations are stored entirely on your machine. Nothing is sent to external servers unless you explicitly enable WebDAV cloud sync.
</Note>
