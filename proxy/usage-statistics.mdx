title: "Track API usage, tokens, and costs across CLI tools"

The usage statistics panel aggregates every API request that passes through CC Switch and turns it into actionable data: how many tokens you consumed, how much it cost, which providers had failures, and how latency varied over time. Starting from v3.13.0, usage data is collected from two independent sources — the proxy request log and direct CLI session logs — so you can track spending even without enabling the proxy. This page explains how to set up cost tracking, what the dashboard shows, and how to configure custom per-model pricing.

## Data sources

| Source | Coverage | Requires proxy? |
|---|---|---|
| **Proxy request log** | All requests forwarded through the CC Switch proxy | Yes |
| **CLI session log** (v3.13.0+) | Claude, Codex, and Gemini session history files read directly | No |

- Codex session data uses precise JSONL parsing with normalized model names for consistent pricing lookups
- Gemini session data is synced directly from Gemini CLI session logs
- Claude session data can also be imported from session log files
- The usage panel supports per-app filtering so data from different CLI tools does not mix

## Setting up cost tracking

<Steps>
  <Step title="Enable logging in the proxy panel">
    Go to **Settings → Advanced → Proxy Service** and make sure the **Enable Logging** toggle is on. Logging is enabled by default, but it must be active for requests to appear in the usage dashboard.
  </Step>
  <Step title="Start the proxy and enable app takeover">
    Start the proxy and enable takeover for each app you want to track. See [Local proxy service](/proxy/proxy-service) for details. CLI session logs are collected automatically regardless of this step.
  </Step>
  <Step title="Open the usage dashboard">
    Go to **Settings → Usage**. Select a time range — **Today**, **Last 7 Days**, or **Last 30 Days** — and pick an app filter if you want to isolate one CLI tool.
  </Step>
  <Step title="Configure model pricing">
    Go to **Settings → Advanced → Pricing Configuration**. Add or edit per-model prices so that cost estimates reflect your actual rates. See [Custom pricing](#custom-pricing) below.
  </Step>
</Steps>

<Note>
  If the request log is empty, check that the proxy service is running, app takeover is enabled for at least one app, and the **Enable Logging** toggle is on in the proxy panel.
</Note>

## Usage dashboard

### Summary cards

The top of the usage page shows four key metrics for the selected time range:

| Metric | Description |
|---|---|
| Total Requests | Number of API requests in the period |
| Total Tokens | Combined input and output token count |
| Estimated Cost | Cost calculated from your pricing configuration |
| Success Rate | Percentage of requests that returned a successful response |

### Trend charts

Two charts visualize usage over time:

**Request trend** — a line chart of request counts by hour (for Today) or by day (for 7 Days / 30 Days). Supports zoom and drag to inspect specific windows.

**Token trend** — stacked token usage with a cost overlay:

| Series | Color | Description |
|---|---|---|
| Input Tokens | Blue | Prompt content sent in each request |
| Output Tokens | Green | Response content generated by the model |
| Cache Creation Tokens | Orange | Tokens charged when creating a prompt cache (typically 1.25× input price) |
| Cache Hit Tokens | Purple | Tokens saved by reusing a prompt cache (typically 0.1× input price) |
| Cost | Red dashed line (right axis) | Estimated cost for the period |

## Detailed data tabs

Three tabs at the bottom of the usage page provide record-level detail.

### Request logs

A row-per-request log with the following fields:

| Field | Description |
|---|---|
| Time | Timestamp of the request |
| App | Claude / Codex / Gemini |
| Provider | Provider name used for this request |
| Model | Model identifier (the billing model) |
| Input Tokens | Input token count |
| Output Tokens | Output token count |
| Cache Read | Cache hit token count |
| Cache Creation | Cache creation token count |
| Total Cost | Estimated cost in USD |
| Latency | Total duration, time to first token, and stream type |
| Status | HTTP status code |

The **Latency** column shows color-coded badges:

| Badge | Color | Threshold |
|---|---|---|
| Total Duration | Green | ≤ 5 s |
| Total Duration | Orange | 5–120 s |
| Total Duration | Red | > 120 s |
| First Token | Same rules | Time to first byte in streaming requests |
| Stream / Non-stream | Blue / Purple | Request type |

Click any row to expand full request parameters, a response content summary, and the error message if the request failed.

You can filter the log by:

| Filter | Options |
|---|---|
| App type | All / Claude / Codex / Gemini |
| Status code | All / 200 / 400 / 401 / 429 / 500 |
| Provider | Text search |
| Model | Text search |
| Time range | Start and end datetime picker |

### Provider statistics

Aggregated per provider:

| Field | Description |
|---|---|
| Provider | Provider name |
| Requests | Total requests routed through this provider |
| Successes | Number of successful responses |
| Failures | Number of failed responses |
| Success Rate | Success percentage |
| Total Tokens | Total token usage |
| Estimated Cost | Cost attributed to this provider |

### Model statistics

Aggregated per model:

| Field | Description |
|---|---|
| Model | Model name |
| Requests | Total requests for this model |
| Input Tokens | Total input tokens |
| Output Tokens | Total output tokens |
| Avg Latency | Average response time |
| Estimated Cost | Cost attributed to this model |

## Custom pricing

CC Switch ships with preset official prices for common models (Claude, GPT, Gemini, DeepSeek, Kimi, and others). If you use a proxy service or reseller with different rates, you can override any model's price.

Go to **Settings → Advanced → Pricing Configuration**.

### Pricing fields

| Field | Description |
|---|---|
| Model ID | Normalized model identifier (see normalization rules below) |
| Display Name | Custom label shown in the usage dashboard |
| Input Price | Price per million input tokens (USD or provider currency) |
| Output Price | Price per million output tokens |
| Cache Read Price | Price per million cache hit tokens |
| Cache Creation Price | Price per million cache creation tokens |

### Model ID normalization

Before looking up a price, CC Switch normalizes the raw model ID from the request:

- Removes everything before the last `/`
- Removes everything after `:`
- Replaces `@` with `-`

Enter the **normalized** ID when adding a pricing entry, not the raw model name from the request.

| Raw model name | Normalized ID to enter | Note |
|---|---|---|
| `stepfun-ai/step-3.5-flash` | `step-3.5-flash` | Provider prefix removed |
| `moonshotai/kimi-k2-0905:exa` | `kimi-k2-0905` | Prefix and `:` suffix removed |
| `gpt-5.2-codex@low` | `gpt-5.2-codex-low` | `@` replaced with `-` |

### Managing pricing entries

- **Add** — Click **Add** and fill in the model ID and prices
- **Edit** — Click the edit icon at the end of any row to modify an existing entry
- **Delete** — Click the delete icon to remove a model's pricing entry

## Model test and stream check

CC Switch can send a lightweight test request to any provider to measure endpoint latency and verify API key validity. This is also used internally by the proxy for health checks on providers in the failover queue.

Go to **Settings → Advanced → Model Test Config** to configure the test model and parameters per app.

### Test coverage

Starting from v3.13.0, stream checks cover all five supported apps:

| App | Notes |
|---|---|
| Claude | Sends a short prompt using the configured Claude test model |
| Codex | Uses the configured Codex test model |
| Gemini | Uses the configured Gemini test model |
| OpenCode | Auto-detected via npm package mapping |
| OpenClaw | Covers all protocol variants (including `openai-completions`), custom `auth-header` detection, and Bedrock error message handling |

### Test parameters

| Parameter | Description | Default | Range |
|---|---|---|---|
| Timeout | Maximum wait per request | 45 s | 10–120 s |
| Max Retries | Retries on failure | 2 | 0–5 |
| Degradation Threshold | Response time above which a provider is marked degraded but still usable | 6000 ms | 1000–30000 ms |

### Test results

| Status | Description |
|---|---|
| Healthy (green) | Response received within the degradation threshold |
| Degraded (yellow) | Response received, but latency exceeded the degradation threshold |
| Unavailable (red) | Request failed or timed out |

Each result shows response latency in milliseconds, time to first byte (TTFB), and an error message when applicable.

<Note>
  Model tests consume a small amount of API quota. Use low-cost models like Claude Haiku, Codex mini variants, or Gemini Flash for testing to minimize costs.
</Note>
